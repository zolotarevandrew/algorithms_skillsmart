1) Разделяем на 3 интерфейса, преонбординг сервис 
- IPreOnboardingService используется вместе в процессе преонбординга (начало конец и получение текущего статуса)
здесь не получится использовать более обобщенную версию.

- IPreOnboardingParamSourceGetter
SRP - выделил отдельный интерфейс для получения преонбординг параметров.
Обобщенность - убрал лишний метод получения конкретного параметра по типу, обобщил в текущем методе.

- IPreOnboardingParamSaver, 
SRP - выделил отдельный интерфейс для добавления преонбординг параметров.
Обобщенность - добавил generic, чтобы можно было добавлять любой специфичный класс параметра, без неявного каста к базовому классу.

2) Интерфейс получения данных шага заставляет реализовывать сразу 3 метода.
При этом эти 3 метода всегда вместе вызываются при получении данных шага.

SRP - отдельный интерфейс для получения данных.
Обобщенность - обьединяем 3 метода одним выходным контрактом, и обьявляем его generic-ом.

3) PdfReaderService - используется внутри конкретных реализаций и есть два отдельных кастомных интерфейса под каждую модель pdf.

SRP - оставляем отдельный интерфейс для парсинга моделей PDF.
Обобщенность - обьединяем 3 интерфейса в один, и делаем его generic-ом.

Итого - ISP по сути помогает в снижении coupling, при этом мелкие интерфейсы не всегда оправданы.
Иногда лучше иметь набор связанных между собой методов одним интерфейсом.
При этом вместе с полиморфностью данный принцип позволяет охватить все больше граней системы.
